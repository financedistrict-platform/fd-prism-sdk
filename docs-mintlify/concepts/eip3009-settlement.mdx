## What is EIP-3009?

**EIP-3009** is an Ethereum Improvement Proposal that enables **gasless token transfers** - users sign an off-chain authorization, and someone else (Facilitator) pays the gas to execute it on-chain.

## The Problem EIP-3009 Solves

### Traditional ERC-20 Transfer

```solidity
// User must have ETH for gas
function transfer(address to, uint256 amount) external {
    _transfer(msg.sender, to, amount);
}
```

**Requirements:**

- User needs USDC (or token being transferred)
- User needs ETH (or native gas token)
- User submits transaction and pays gas

**Problems:**

- **Onboarding friction** - New users need two tokens
- **UX complexity** - Must explain gas concept
- **Cost** - User pays gas even for small payments

### EIP-3009 Solution

```solidity
// User only needs to sign, Facilitator pays gas
function transferWithAuthorization(
    address from,
    address to,
    uint256 value,
    uint256 validAfter,
    uint256 validBefore,
    bytes32 nonce,
    uint8 v,
    bytes32 r,
    bytes32 s
) external {
    // Verify signature
    bytes32 hash = keccak256(abi.encode(...));
    address signer = ecrecover(hash, v, r, s);
    require(signer == from, "Invalid signature");

    // Execute transfer
    _transfer(from, to, value);
}
```

**Advantages:**

- **No gas required** - User only signs message
- **Better UX** - Single token needed
- **Lower cost** - Facilitator can batch transactions
- **Flexibility** - Authorization can be used later

    function transferWithAuthorization(...) external {
        // 1. Build EIP-712 hash

        address signer = ECDSA.recover(hash, v, r, s);

        require(block.timestamp <= validBefore, "Expired");

## EIP-712 Typed Data Structure

### Domain Separator

**Prevents cross-chain/cross-contract replay attacks**

```typescript
{
  name: "USD Coin",
  version: "2",
  chainId: 84532,              // base-sepolia
  verifyingContract: "0x036CbD53842c5426634e7929541eC2318f3dCF7e"
}
```

**Why needed:**

- Same signature on Ethereum would be invalid on Base
- Same signature for Token A would be invalid for Token B

### TransferWithAuthorization Type

```typescript
{
  TransferWithAuthorization: [
    { name: "from", type: "address" },
    { name: "to", type: "address" },
    { name: "value", type: "uint256" },
    { name: "validAfter", type: "uint256" },
    { name: "validBefore", type: "uint256" },
    { name: "nonce", type: "bytes32" },
  ];
}
```

## Signature Format

### ECDSA Signature Components

```
Signature = r + s + v
- r: 32 bytes (x-coordinate of R)
- s: 32 bytes (signature proof)
- v: 1 byte (recovery id: 27 or 28)
```

**Example:**

```
0xabcdef1234567890abcdef1234567890abcdef1234567890abcdef1234567890  // r (32 bytes)
  1234567890abcdef1234567890abcdef1234567890abcdef1234567890abcdef  // s (32 bytes)
  1b                                                                // v (1 byte = 27)
```

### Signature Recovery (ecrecover)

```solidity
// On-chain signature verification
function recoverSigner(
    bytes32 hash,
    uint8 v,
    bytes32 r,
    bytes32 s
) internal pure returns (address) {
    return ecrecover(hash, v, r, s);
}
```

**Usage:**

```solidity
bytes32 hash = _hashTypedDataV4(structHash);
address signer = ecrecover(hash, v, r, s);
require(signer == from, "Invalid signature");
```

## Security Features

<AccordionGroup>
  <Accordion icon="fingerprint" title="Nonce Uniqueness">
    **Purpose:** Prevent replay attacks and double-spending
    
    **Implementation:**
    ```solidity
    mapping(address => mapping(bytes32 => bool)) public nonceUsed;
    
    function transferWithAuthorization(...) external {
        require(!nonceUsed[from][nonce], "Nonce already used");
        nonceUsed[from][nonce] = true;
        // ... rest of execution
    }
    ```
    
    **Nonce Generation:**
    ```typescript
    // Gateway generates cryptographically secure random bytes
    const nonce = ethers.randomBytes(32);
    // "0x7d2c3a1e4f5b6c8d9e0f1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2"
    ```
    
    **Why 32 bytes?**
    - 2^256 possible values (astronomically large)
    - Collision probability: negligible
    - Same security as Ethereum addresses
  </Accordion>
  
  <Accordion icon="clock" title="Time-Bound Authorizations">
    **Purpose:** Prevent stale quotes and front-running
    
    **validAfter:**
    - Earliest timestamp authorization can execute
    - Usually `0` (immediate) or `block.timestamp`
    - Prevents premature execution
    
    **validBefore:**
    - Latest timestamp authorization can execute
    - Typical: 5-15 minutes from quote generation
    - Prevents indefinite validity
    
    **Example:**
    ```typescript
    const now = Math.floor(Date.now() / 1000);
    const quote = {
      validAfter: now,
      validBefore: now + 900  // 15 minutes
    };
    ```
    
    **On-Chain Validation:**
    ```solidity
    require(
        block.timestamp >= validAfter,
        "Authorization not yet valid"
    );
    require(
        block.timestamp <= validBefore,
        "Authorization expired"
    );
    ```
  </Accordion>
  
  <Accordion icon="shield" title="Domain Separation">
    **Purpose:** Prevent cross-chain and cross-contract replay
    
    **Domain Hash:**
    ```solidity
    bytes32 domainSeparator = keccak256(abi.encode(
        keccak256("EIP712Domain(string name,string version,uint256 chainId,address verifyingContract)"),
        keccak256(bytes("USD Coin")),
        keccak256(bytes("2")),
        84532,  // base-sepolia
        address(this)
    ));
    ```
    
    **Why Different Chains Can't Replay:**
    1. Signature on Ethereum has `chainId: 1`
    2. Signature on Base has `chainId: 84532`
    3. Domain hashes are different
    4. Signature verification fails on wrong chain
    
    **Why Different Contracts Can't Replay:**
    1. Contract A has `verifyingContract: 0xAAA...`
    2. Contract B has `verifyingContract: 0xBBB...`
    3. Domain hashes are different
    4. Signature verification fails on wrong contract
  </Accordion>
  
  <Accordion icon="user-check" title="Recipient Lock">
    **Purpose:** Prevent payment redirection attacks
    
    **Signed Message Includes Recipient:**
    ```json
    {
      "from": "0xClient...",
      "to": "0xProvider...",    // ← Recipient locked in signature
      "value": "1000000",
      ...
    }
    ```
    
    **Attack Scenario (Without Recipient Lock):**
    1. Attacker intercepts signed authorization
    2. Changes `to` address to their own
    3. Submits modified authorization
    4. Steals payment
    
    **Defense (With Recipient Lock):**
    1. Attacker intercepts authorization
    2. Tries to change `to` address
    3. Signature verification fails (hash mismatch)
    4. Transaction reverts
  </Accordion>
  
  <Accordion icon="bolt" title="Front-Running Mitigation">
    **Problem:**
    - Attacker monitors mempool
    - Sees pending `transferWithAuthorization` transaction
    - Submits same transaction with higher gas
    - Gets mined first
    
    **Defense 1: Private Mempool**
    - Facilitator uses Flashbots/Eden Network
    - Transaction bypasses public mempool
    - Directly included by block builder
    
    **Defense 2: Nonce Protection**
    - Even if front-run, only first transaction succeeds
    - Second transaction reverts (nonce already used)
    - Original payer protected from double-charge
    
    **Defense 3: Recipient Lock**
    - Front-runner can't change recipient
    - Payment still goes to intended provider
  </Accordion>
</AccordionGroup>

## Gas Cost Analysis

<Tabs>
  <Tab title="Traditional ERC-20 Transfer">
    ```solidity
    function transfer(address to, uint256 amount) external {
        _transfer(msg.sender, to, amount);
    }
    ```
    
    **Gas Cost:** ~50,000 gas
    
    **At 50 gwei gas price + $3,000 ETH:**
    - Cost: 50,000 × 50 × 10^-9 × $3,000 = **$7.50**
    
    **Paid by:** User (requires ETH)
  </Tab>
  
  <Tab title="EIP-3009 Authorization">
    ```solidity
    function transferWithAuthorization(...) external {
        // Signature recovery + validation + transfer
    }
    ```
    
    **Gas Cost:** ~80,000 gas (30k more for signature verification)
    
    **At 50 gwei gas price + $3,000 ETH:**
    - Cost: 80,000 × 50 × 10^-9 × $3,000 = **$12.00**
    
    **Paid by:** Facilitator (not user!)
    
    **Facilitator Fee:** ~1-2% of payment amount
    - For $1 payment: $0.01-$0.02 fee
    - Still cheaper than requiring users to have ETH
  </Tab>
  
  <Tab title="Batch Processing">
    **Facilitator can batch multiple authorizations:**
    
    ```solidity
    function batchTransferWithAuthorization(
        Authorization[] calldata auths
    ) external {
        for (uint i = 0; i < auths.length; i++) {
            _transferWithAuthorization(auths[i]);
        }
    }
    ```
    
    **Gas Cost:**
    - First authorization: 80,000 gas
    - Each additional: ~60,000 gas (20k savings from warm storage)
    
    **10 authorizations:**
    - Total: 80,000 + (9 × 60,000) = 620,000 gas
    - Average: 62,000 gas per authorization
    
    **Cost per authorization:** $9.30 (vs $12.00 individual)
    
    **Savings:** 22.5% through batching
  </Tab>
</Tabs>

    import "@openzeppelin/contracts/token/ERC20/ERC20.sol";
    import "@openzeppelin/contracts/utils/cryptography/ECDSA.sol";
    import "@openzeppelin/contracts/utils/cryptography/EIP712.sol";
    
    contract USDCWithAuthorization is ERC20, EIP712 {
        using ECDSA for bytes32;
        
        bytes32 public constant TRANSFER_WITH_AUTHORIZATION_TYPEHASH =
            keccak256(
                "TransferWithAuthorization(address from,address to,uint256 value,uint256 validAfter,uint256 validBefore,bytes32 nonce)"
            );
        
        mapping(address => mapping(bytes32 => bool)) public nonceUsed;
        
        constructor() ERC20("USD Coin", "USDC") EIP712("USD Coin", "2") {}
        
        function transferWithAuthorization(
            address from,
            address to,
            uint256 value,
            uint256 validAfter,
            uint256 validBefore,
            bytes32 nonce,
            uint8 v,
            bytes32 r,
            bytes32 s
        ) external {
            // Build EIP-712 hash
            bytes32 structHash = keccak256(
                abi.encode(
                    TRANSFER_WITH_AUTHORIZATION_TYPEHASH,
                    from,
                    to,
                    value,
                    validAfter,
                    validBefore,
                    nonce
                )
            );
            bytes32 hash = _hashTypedDataV4(structHash);
            
            // Recover signer
            address signer = hash.recover(v, r, s);
            require(signer == from, "Invalid signature");
            
            // Validate authorization
            require(block.timestamp >= validAfter, "Not yet valid");
            require(block.timestamp <= validBefore, "Expired");
            require(!nonceUsed[from][nonce], "Nonce already used");
            
            // Mark nonce as used
            nonceUsed[from][nonce] = true;
            
            // Execute transfer
            _transfer(from, to, value);
            
            emit TransferWithAuthorization(from, to, value, nonce);
        }
        
        event TransferWithAuthorization(
            address indexed from,
            address indexed to,
            uint256 value,
            bytes32 indexed nonce
        );
    }

## Comparison with Other Standards

| Feature            | EIP-3009               | EIP-2612 (Permit)   | EIP-4337 (Account Abstraction) |
| ------------------ | ---------------------- | ------------------- | ------------------------------ |
| **Gas sponsor**    | Yes (Facilitator)      | No (user pays)      | Yes (Paymaster)                |
| **Token standard** | ERC-20 extension       | ERC-20 extension    | Works with any token           |
| **Complexity**     | Low                    | Low                 | High                           |
| **Adoption**       | USDC, USDT             | Many ERC-20s        | Growing                        |
| **Use case**       | Payment gateways       | DeFi approvals      | General purpose                |
| **Nonce tracking** | Per-authorization      | Per-address counter | Complex nonce schemes          |
| **Expiry**         | validAfter/validBefore | deadline            | Flexible                       |

<Info>
  **Prism uses EIP-3009** because it's battle-tested (USDC, USDT), simple to
  implement, and purpose-built for payment use cases.
</Info>
